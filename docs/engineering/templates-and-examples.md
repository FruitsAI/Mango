# 模板与示例规范

## 一、文档目的

模板不是“方便复制”的附件，而是把规范真正变成默认路径的手段。

## 二、当前模板目录

```text
tooling/generators/templates/
  renderer-feature/
  desktop-ipc-module/
  api-module/
  worker-job/
  openapi-spec/
  sqlite-migration/
```

## 三、模板使用规则

- 新 feature 优先从模板起步，而不是从旧文件复制
- 模板必须体现当前规范目录，而不是历史目录
- 模板更新时，必须同步更新本说明文档
- 契约模板和 migration 模板必须与对应规范文档同步更新

## 四、必须包含的内容

每个模板至少要说明：

- 目录结构
- 文件职责
- 依赖边界
- 必要测试
- 评审重点
- 命名规范与验证方式

## 五、推荐做法

- 用模板描述“最小可工作的结构”
- 用示例标明主流程文件应该放哪里
- 让模板与 CI 门禁、contracts、shared config 同步演进
- 对 OpenAPI、SQLite migration 这类高约束内容提供可直接复制的样板文件

## 六、禁止事项

- 模板过时但无人维护
- 模板里继续引用已废弃目录
- 用模板生成一堆 TODO 文件却不说明职责
- 契约模板和真实错误模型、分页模型长期漂移

## 七、评审清单

- 模板是否符合最新目录规范
- 模板是否体现当前 contracts / core / adapters 边界
- 模板是否包含验证建议
- 文档与模板是否一致
- 是否覆盖 OpenAPI 与 migration 这类高风险重复工作

## 八、常见反模式

- 复制旧功能目录再“稍微改改”
- 模板只有文件名，没有职责说明
- 模板和真实工程长期漂移
- 契约和 SQL 每次都从空白开始写，导致风格不断分叉
