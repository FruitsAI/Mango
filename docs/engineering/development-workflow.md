# 研发工作流

## 一、目标

这份文档用于统一 Mango 的研发推进方式，避免出现“大家都在忙，但节奏混乱”的情况。

## 二、推荐流程

### 第一步：确认目标文档

所有开发任务必须能指向某个正式文档，例如：

- PRD
- 路线图
- 需求池
- 技术设计
- 测试策略

### 第二步：拆分任务

拆任务时要明确：

- 目标是什么
- 不做什么
- 验证方式是什么
- 影响哪些模块

### 第三步：先定义验证方式

在实现前先回答：

- 用什么测试证明改动有效
- 用什么命令证明类型与构建没坏
- 需要回归哪些场景
- 是否需要跑 `pnpm architecture-check`

### 第四步：最小化实现

优先写最小可工作的版本，不把未来能力一并做掉。

### 第五步：同步文档

任何影响公共行为的改动，都必须同步更新文档。

### 第六步：统一走根命令

默认使用以下命令作为工程验证入口：

- `pnpm lint`
- `pnpm architecture-check`
- `pnpm test`
- `pnpm typecheck`
- `pnpm build`
- `pnpm smoke:desktop`
- `pnpm verify`

## 三、任务拆解建议

一项开发任务最好拆成：

1. 目标确认
2. 测试准备
3. 最小实现
4. 本地验证
5. 文档更新
6. 合并准备

## 四、定义“完成”的最低标准

一项任务只有在以下条件都满足时才算完成：

- 代码实现已落地
- 测试已通过
- 类型检查已通过
- 构建已通过
- 必要文档已更新
- 必要边界检查已通过

## 五、哪些情况必须停下来重新对齐

- 发现需求不明确
- 发现实现方案与文档冲突
- 发现范围明显扩大
- 发现当前模块边界不再适用

## 六、禁止事项

- 没有目标文档就直接开发
- 需求变化但文档不更新
- 没有验证就声称完成
- 一次提交同时包含多个无关目标
- 绕开根命令体系只跑局部私有脚本
