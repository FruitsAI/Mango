# 技术架构总览

## 一、架构目标

Mango 的技术架构必须同时满足以下要求：

- 支持桌面端本地执行场景
- 保持主流程可见可控
- 能够替换底层 Agent 实现
- 让 UI、领域逻辑、适配层边界清楚
- 能支撑后续逐步从 mock 走向真实执行

## 二、仓库结构

当前仓库采用 `pnpm + turbo` Monorepo，按“应用入口 / 共享包 / 工具与测试 / 文档与基础设施”分层组织。

### `apps/desktop`

桌面端应用层，负责：

- Electron 主进程
- Preload 安全桥
- React 渲染层
- 本地状态协调

### `apps/web` / `apps/api` / `apps/worker`

未来预留入口，分别承载：

- Web 工作台或官网前端
- 云后端 API
- 异步任务与后台执行器

### `packages/core`

领域核心层，负责：

- 任务状态定义
- 状态转换规则
- 权限模型
- 执行事件模型
- 工作区模型

### `packages/adapters`

外部能力接入层，负责：

- 检测外部 agent / CLI 是否可用
- 生成执行计划
- 在批准后返回执行事件

### `packages/contracts`

共享契约层，负责：

- IPC channel 名称
- 桌面 DTO
- 未来 REST / OpenAPI 相关 DTO
- 统一错误模型

### `packages/ui`

共享 UI 资产层，负责：

- Design token
- 未来桌面 / Web 复用组件资产

### `packages/config-*`

共享工程配置层，负责：

- TypeScript 配置
- ESLint Flat Config
- Vitest 配置
- Playwright 配置预留

## 三、运行时分层

### 1. 主进程

主进程是 Mango 的运行协调中心，适合承载：

- 本地持久化
- 与系统能力交互
- 任务编排
- 与 Adapter 的交互

### 2. Preload

Preload 负责暴露一层安全、收敛、类型化的 API 给渲染层。它的职责不是实现业务，而是约束边界。

### 3. 渲染层

渲染层负责：

- 展示状态
- 收集用户输入
- 触发明确的交互动作
- 呈现计划、权限、执行、回顾

## 四、核心执行链路

1. 应用启动
2. 主进程加载本地状态
3. 主进程检测当前工作区与 adapter 状态
4. 渲染层展示工作台
5. 用户发起任务
6. 主进程调用 adapter 生成计划
7. 用户批准后，主进程调用 adapter 执行
8. 事件流写入状态并同步到界面
9. 任务结束后生成 review

## 五、为什么不用“所有逻辑都放前端”

因为 Mango 的关键价值在于本地能力协调和安全边界控制。如果把执行协调逻辑放在前端：

- 结构会混乱
- 安全边界会变弱
- 后续接入真实执行器会更困难

## 六、为什么当前不做更重的服务端架构

当前 Mango 的验证重点在桌面本地闭环，而不是云端协作。因此此阶段不引入额外后端，是为了：

- 降低复杂度
- 加快产品验证
- 把注意力放在主体验上

## 七、架构演进预留点

- 从单 adapter 到多 adapter
- 从 JSON 到 SQLite
- 从静态事件流到可恢复会话
- 从本地单用户到未来可选的远端服务能力
