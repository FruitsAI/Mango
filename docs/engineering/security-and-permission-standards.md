# 安全与权限规范

## 一、适用范围

适用于 Electron 安全、密钥与环境变量、文件系统与 Shell 权限、网络能力、浏览器自动化能力以及本地日志脱敏。

## 二、必须遵守的规则

### 1. Electron 基线

- 开启 `contextIsolation`
- 关闭 `nodeIntegration`
- preload 只暴露最小 API 面
- renderer 不直接调用 Node / Electron 能力

### 2. 权限模型

- `shell`、`filesystem`、`network`、`browser` 都是受控能力
- 风险操作必须经过 `PermissionPolicy`
- 权限结果必须可见、可回顾、可在任务记录中留痕

### 3. 环境变量与密钥

- 环境变量必须有白名单
- 敏感值不得写入日志、错误堆栈、任务总结
- 本地密钥或 token 不允许硬编码进仓库

### 4. 本地存储

- 持久化层统一收口
- 存储内容最小化
- 未来迁移 SQLite 时保持同样权限与脱敏原则

## 三、推荐做法

- 在 contracts 与文档中明确高风险错误码
- 在主进程做二次校验，不依赖前端文案
- 在 review 中展示权限申请与执行结果
- 为网络与浏览器能力增加更高风险提示

## 四、禁止事项

- preload 暴露宽泛系统 API
- 把命令执行能力直接透传给 renderer
- 在 README、示例代码、测试里硬编码真实密钥
- 日志中输出完整 token、cookie、路径敏感信息

## 五、测试要求

- 权限允许与拒绝至少各有一条验证路径
- 安全边界改动必须验证 renderer 仍无法直接调用受限能力
- 新增环境变量必须验证缺失与非法值处理

## 六、评审清单

- 是否扩大了 renderer 能力面
- 是否新增了高风险能力但未加入权限模型
- 是否存在敏感信息落日志
- 是否同步更新了隐私、遥测或安全文档

## 七、常见反模式

- 为了图方便在 preload 里暴露整个 Electron 模块
- 把权限批准只当 UI 弹窗，而不是可审计事件
- 在调试日志里输出完整命令和敏感配置
