# 信息架构

## 一、目标

Mango 的信息架构必须服务一个核心问题：

> 用户在任何时刻都应该知道：当前在哪个工作区、当前任务到了哪一步、接下来是否需要自己做决定。

## 二、一级结构

### 左侧栏

- 品牌信息
- 环境信息
- 工作区列表

### 主内容区

- Hero 区与当前状态
- 执行主流程展示
- 任务输入区
- 计划区
- 执行时间线
- 结果回顾区

## 三、核心对象

Mango 的 UI 不是围绕“聊天消息”设计，而是围绕以下对象设计：

- 工作区 `Workspace`
- 任务会话 `TaskSession`
- 计划 `TaskPlan`
- 权限请求 `PermissionRequest`
- 执行事件 `ExecutionEvent`
- 任务回顾 `TaskReview`

## 四、对象关系

- 一个工作区可以关联多个任务会话
- 一个任务会话在任意时间只有一个主状态
- 一个任务会话可以拥有一个计划
- 一个计划包含多个步骤和权限请求
- 一个任务会话在执行中会产生多个执行事件
- 一个任务会话结束后生成一个回顾摘要

## 五、主流程结构

### 1. 任务输入

用户输入目标，不是输入零散命令。

### 2. 计划展示

用户看到 Mango 准备如何做。

### 3. 权限批准

用户决定是否允许 Mango 执行关键动作。

### 4. 执行事件展示

用户看到 Mango 正在做什么，而不是等它“做完告诉你”。

### 5. 结果总结

用户能判断这次任务是否值得继续。

## 六、信息优先级

### 第一优先级

- 当前任务状态
- 当前工作区
- 当前计划
- 当前权限请求

### 第二优先级

- 执行事件流
- 当前 adapter 状态
- 最近任务历史

### 第三优先级

- 版本信息
- 遥测与设置
- 帮助与反馈入口

## 七、设计约束

- 不允许把关键执行状态藏进二级弹层
- 不允许把权限提示做成看不懂的技术文案
- 不允许让用户在多个地方找同一类关键信息
- 不允许任务执行完成后只留下原始日志，没有结构化结论
