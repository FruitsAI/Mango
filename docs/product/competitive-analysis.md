# 竞品分析

## 一、分析目的

竞品分析不是为了复制别人，而是为了明确 Mango 应该借鉴什么、放弃什么、坚持什么。

## 二、分析对象

本轮重点参考以下三个项目：

- AionUi
- CodePilot
- MossX

## 三、分析维度

我们只看最影响 Mango 的三类能力：

1. 会话工作台
2. 任务执行
3. 桌面集成

## 四、AionUi

### 优势

- 多 Agent 与工作空间概念较强
- 整体产品想象空间大
- 自动化与扩展方向丰富

### 对 Mango 的启发

- Mango 后续可以采用更清晰的多工作区视角
- 应该把“Agent 是一个系统能力”而非“一个聊天对象”做出来

### Mango 不应该照搬的部分

- 在 v1 过早扩大目标人群
- 在没有打磨核心闭环前就做大而全的自动化

## 五、CodePilot

### 优势

- 明显面向开发者
- 更贴近本地开发流程
- GUI + 本地 agent 的方向清晰

### 对 Mango 的启发

- Mango 应该坚持“桌面开发者工具”的身份
- 不应该把界面做成单纯的壳，而应做成真正的控制台

### Mango 需要做得更好的地方

- 把计划和权限展示得更清楚
- 把执行与复盘做成产品亮点，而不只是附属视图

## 六、MossX

### 优势

- 面向开发者的产品定位较明确
- 有多引擎扩展思路
- 适合作为未来适配器体系的方向参考

### 对 Mango 的启发

- Mango 的适配器体系需要从第一天就有边界
- 未来可以支持多个底层 Agent 或模型，但不急于在 v1 做全

### Mango 需要避免的风险

- 在尚未证明主流程价值前就堆太多集成能力
- 把“扩展性”误当成“产品完成度”

## 七、Mango 的差异化结论

Mango 当前最应该强调的不是：

- 模型数量
- 自动化广度
- 花哨 UI

而是：

- 可见计划
- 可见权限
- 可见执行
- 可见结果

## 八、对产品策略的影响

基于竞品观察，Mango 当前策略应保持：

- 聚焦开发者，而不是全人群
- 聚焦工作台，而不是聊天产品
- 聚焦一条高可信执行路径，而不是一堆半成品能力
