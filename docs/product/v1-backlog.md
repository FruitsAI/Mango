# v1 需求池与优先级

## 一、使用方式

本文件不是“想法收纳箱”，而是 Mango v1 阶段的正式需求池。每项需求都要能说明：

- 解决什么问题
- 服务哪个主场景
- 为什么现在做
- 不做的代价是什么

## 二、P0：必须完成

### 1. 真实 CLI Adapter 接入

- 目标：替代 mock adapter，形成真实执行能力
- 原因：没有真实执行，产品核心价值无法验证

### 2. 工作区管理增强

- 目标：支持最近工作区、切换工作区、工作区状态刷新
- 原因：开发者任务天然依赖工作区上下文

### 3. 任务恢复与失败重试

- 目标：在失败或中断后可以继续，而不是一切归零
- 原因：真实桌面任务不可能永远一次成功

### 4. 权限体验完善

- 目标：把权限批准从“展示”升级为“真正的控制点”
- 原因：这是 Mango 的信任核心

### 5. 结果回顾增强

- 目标：增加更明确的事件分组、文件变化说明、最终结论区
- 原因：用户要快速判断结果是否可接受

## 三、P1：强烈建议完成

### 1. SQLite 持久化

- 目标：替代当前 JSON 存储
- 原因：便于后续任务历史、索引与恢复能力扩展

### 2. 安装与更新闭环

- 目标：补齐 Beta 所需安装包、更新策略与版本通道

### 3. 反馈与问题采集入口

- 目标：让试用用户反馈路径清晰可控

### 4. 首次引导

- 目标：帮助用户理解工作区、权限、执行流程

## 四、P2：可延后

### 1. 多 adapter 支持

- 在单 adapter 跑通前不优先

### 2. 更复杂的浏览器自动化

- 在开发者主流程验证前不优先

### 3. 团队协作能力

- 明确不是 v1 目标

## 五、优先级判断规则

当有新需求加入时，使用以下规则：

1. 是否强化主闭环
2. 是否服务主用户群
3. 是否提升信任与可控性
4. 是否会影响 v1 节奏

如果前三项不明显成立，而第四项明显拖慢节奏，则默认不进入 P0/P1。
