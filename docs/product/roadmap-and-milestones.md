# 路线图与里程碑

## 一、文档目的

本文件定义 Mango 从当前骨架阶段到公测阶段的推进顺序，用来约束研发节奏，防止需求不断扩张。

## 二、阶段划分原则

每个阶段只解决一个主问题，不同时解决多个大问题。

## 三、M0：产品定义完成

### 目标

完成一整套可执行文档，确保后续开发不再依赖口头记忆。

### 交付物

- 产品定位与愿景
- PRD
- 用户画像与场景
- 竞品分析
- 信息架构
- 技术设计
- 路线图与 backlog

### 完成标准

- 团队可以用文档回答“为什么做、做给谁、先做什么、不做什么”

## 四、M1：可运行骨架

### 目标

建立跨平台桌面应用骨架与核心领域模型。

### 交付物

- Electron + React + TypeScript Monorepo
- Core 状态机与权限模型
- Mock adapter
- 最小持久化
- 初步 UI 工作台

### 完成标准

- 可以本地运行、测试、构建

## 五、M2：真实执行闭环

### 目标

把 mock 执行替换为真实 CLI Agent 执行。

### 关键工作

- 接入真实 CLI adapter
- 增强错误处理与失败恢复
- 增加任务历史与恢复能力
- 把权限批准变成真正的执行前闸门

### 完成标准

- 用户可以在真实项目中完成一轮端到端任务

## 六、M3：Beta 可用性打磨

### 目标

从“能跑”变成“愿意给别人试用”。

### 关键工作

- 完善工作区管理
- 强化回顾与差异展示
- 完善安装、更新、反馈链路
- 补齐跨平台验证

### 完成标准

- 首批试用用户可以稳定完成主流程，不频繁卡住

## 七、M4：公开测试

### 目标

开始面向社区公开分发。

### 关键工作

- GitHub Releases 节奏建立
- Issue 模板、反馈流程、隐私文档完善
- 公测版本说明、更新节奏、问题优先级机制建立

### 完成标准

- 产品可以被持续安装、反馈、迭代，而不只是单次演示

## 八、阶段推进守则

- 没完成当前阶段定义的问题，就不要跳去下一阶段
- 不允许因为灵感涌现而打断主里程碑
- 所有新增需求都必须明确属于哪个阶段
