# Mango 愿景与定位

## 一、愿景

Mango 的长期愿景不是做“更会聊天的 AI”，而是做“更值得信任的桌面执行伙伴”。

在 Mango 的理想状态里，用户只需要明确目标和边界，系统就可以在可见、可控、可回顾的前提下完成实际工作。

## 二、使命

帮助开发者把高频、重复、碎片化、跨工具的本地任务，交给一个值得信任的桌面 Agent 工作台来完成。

## 三、品牌含义

### 名称来源

`Mango` 对应 `Man, Go!`

- `Man`：用户、人类意图、判断与决策
- `Go`：行动、执行、落地、推进

### 品牌传达

- Mango 不代替用户判断
- Mango 接过执行压力
- Mango 让任务从“想法”变成“结果”

## 四、产品定位

Mango 的定位不是：

- 通用聊天机器人
- 纯 IDE 插件
- 重型流程自动化平台
- 单一模型的套壳应用

Mango 的定位是：

> 一个面向开发者的桌面 Agent 工作台。

## 五、差异化方向

### 差异点 1：桌面执行闭环

Mango 把“计划、授权、执行、复盘”当成一个整体体验设计，而不是零散拼接。

### 差异点 2：权限透明

高风险操作不是后台行为，而是产品层的显式对象。

### 差异点 3：任务结果留痕

每次任务都应当留下结构化结果，而不是只剩下一段聊天记录。

### 差异点 4：开发者工作流优先

工作区、终端、Git、文件变化、日志，是 Mango 的核心战场。

## 六、v1 战略选择

为了避免一开始就失焦，Mango v1 明确采用以下策略：

- 用户聚焦：开发者优先
- 平台聚焦：桌面优先
- 能力聚焦：执行闭环优先
- 商业聚焦：开源优先，商业化后置

## 七、未来演进方向

Mango 后续可以逐步拓展：

- 从 mock adapter 走向真实 CLI adapter
- 从单工作区走向多工作区
- 从任务摘要走向差异对比与任务恢复
- 从开发任务走向部分通用桌面工作流

但所有扩展都不能破坏一个前提：

> Mango 必须始终是“可信执行”，而不是“黑盒自动化”。
