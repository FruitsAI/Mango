# 用户画像与核心场景

## 一、为什么要单独写这份文档

很多项目后期跑偏，不是因为开发能力不够，而是因为团队慢慢忘了“到底是给谁做的”。这份文档的作用，就是把 Mango 的核心服务对象钉住。

## 二、核心用户画像

### 用户画像 A：独立开发者

- 通常一个人负责产品、前端、后端、部署、发布
- 需要快速推进项目
- 愿意尝试效率工具
- 不想被复杂配置或黑盒行为拖慢

### 用户画像 B：全栈工程师

- 在多个项目之间切换
- 经常要处理重复性代码任务、重构、脚本执行、文档补齐
- 更重视任务闭环和结果可信度

### 用户画像 C：技术型产品创作者

- 会自己搭工程与部署
- 更在意“从想法到成品”的效率
- 需要一个能够接管执行但又不失控的工具

## 三、用户共同特征

- 对桌面端工具依赖较高
- 对终端和文件系统并不陌生
- 对误删文件、错误命令、错误改动非常敏感
- 希望减少机械操作，但不接受完全黑箱

## 四、Mango v1 保留的三个核心场景

### 场景 1：本地项目任务执行

用户希望 Mango 基于当前工程上下文，先给出计划，再执行一批明确的本地开发任务。

典型例子：

- 初始化项目骨架
- 补齐缺失文档
- 生成发布清单
- 整理测试计划

### 场景 2：终端、文件、Git 与结果联动

用户希望 Mango 把执行日志、文件变化、工作区状态与结果摘要收拢到一个工作台里。

### 场景 3：任务结束后能复盘

用户希望在执行结束后快速回答这些问题：

- 它做了什么
- 它改了哪些东西
- 它有没有失败
- 我接下来要不要继续让它做

## 五、暂不优先的场景

- 面向非技术用户的办公自动化
- 团队协作审批
- 超长链路的浏览器自动化
- 企业知识库问答

## 六、场景判断标准

后续新增需求时，可以用下面的问题判断是否属于 Mango 的主航道：

1. 这个需求是否发生在开发者本地桌面工作流中？
2. 这个需求是否强化了计划、授权、执行、复盘闭环？
3. 这个需求是否提高了任务可信度，而不是只增加功能数量？
4. 这个需求是否能在 v1 或 v2 的节奏中合理落地？

只要以上问题多数回答为“否”，就应该谨慎进入范围。
