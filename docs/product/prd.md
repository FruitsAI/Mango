# Mango v1 产品需求文档

## 一、文档目的

本文档定义 Mango v1 的目标用户、问题边界、范围取舍、关键流程、成功标准与非目标。后续所有功能开发都应以本文件作为产品范围判断的第一依据。

## 二、产品一句话定义

Mango 是一个面向开发者的桌面端 Agent 工作台，帮助用户把“想做什么”转化为“已审批、可执行、可回顾”的任务闭环。

## 三、产品背景

当前市面上的 Agent 工具大多存在以下问题：

- 更像聊天产品，而不是执行产品
- 在执行高风险操作时缺少清晰的可见授权
- 本地工程上下文、终端输出、文件变化被拆散在不同工具中
- 完成一次任务后缺少结构化复盘结果

Mango 的机会不在于“比别人接更多模型”，而在于让桌面端执行体验更可靠、更透明、更符合开发者日常工作方式。

## 四、目标用户

### 主目标用户

- 独立开发者
- 全栈工程师
- 习惯本地终端、Git、编辑器协作的人

### 用户共性

- 愿意尝试新工具，但对错误执行容忍度低
- 对本地代码、文件和命令执行有高敏感度
- 希望减少机械工作，但不愿意失去控制权

## 五、用户核心问题

### 问题 1：从需求到执行缺少结构化闭环

用户往往只能在聊天框里提出需求，后续执行过程不透明，容易产生不信任。

### 问题 2：高风险操作缺少可见授权

当 Agent 需要访问本地文件、执行 shell、联网时，用户难以判断“它现在到底在做什么”。

### 问题 3：执行结果难以复盘

任务做完后，用户仍然需要自己拼接日志、文件变更、结果总结，才能判断这次执行是否真的达标。

## 六、v1 产品目标

Mango v1 的目标不是做完所有能力，而是证明下面这件事成立：

> 开发者愿意把真实本地任务交给一个桌面 Agent 工作台来规划和执行，前提是流程透明、权限可控、结果可回顾。

## 七、v1 成功标准

### 体验目标

- 新用户首次打开后，10 分钟内能完成一次最小任务闭环
- 用户能在执行前看到结构化计划
- 用户能理解并批准每一类高风险权限
- 用户能在任务结束后看到执行证据和结果摘要

### 业务目标

- 验证 Mango 的核心定位是否有用户吸引力
- 为后续真实 CLI 接入打下产品与工程骨架
- 建立一套足够稳定的文档和研发流程

## 八、v1 范围

### 必做能力

- 首次启动与本地环境检测
- 工作区上下文识别
- 任务输入框与任务会话概念
- 计划生成与计划展示
- 权限展示与显式批准
- 执行事件时间线
- 结果摘要与变更文件列表
- 本地设置和任务历史持久化
- 基础测试、构建、CI、发布文档

### 必须可见的信息

- 当前工作区
- 当前适配器状态
- 当前任务状态
- 本次任务的计划步骤
- 请求的权限种类
- 执行期间事件
- 任务结束摘要

## 九、v1 非目标

以下内容在 v1 明确不做：

- 多人协作与组织账号
- 订阅、支付、许可证
- 云端同步
- 重型浏览器自动化平台
- 自研模型推理引擎
- 大而全的办公自动化能力

## 十、核心用户流程

### 流程 A：首次任务执行

1. 用户启动 Mango
2. Mango 检测当前工作区和环境
3. 用户输入任务目标
4. Mango 生成执行计划
5. Mango 展示权限请求
6. 用户批准后开始执行
7. Mango 展示执行事件流
8. Mango 输出结果摘要和变更信息

### 流程 B：失败任务复盘

1. 用户查看失败任务
2. 用户看到失败原因、执行节点、事件上下文
3. 用户决定是否重试、修改提示、或取消

## 十一、产品原则

### 原则 1：先计划，再执行

不允许把“直接执行”作为主路径。

### 原则 2：权限必须是产品能力，不是技术细节

用户必须看得见、理解得了、愿意批准。

### 原则 3：结果必须能回顾

任务结束不是终点，复盘能力是产品可信度的一部分。

### 原则 4：本地开发者体验优先

先把开发者桌面场景做到顺，再考虑向更多场景扩展。
