# 测试策略

## 一、目标

测试的目的不是为了“凑覆盖率”，而是为了证明 Mango 的主闭环可用、可信、可回归。

## 二、测试分层

### 1. 领域层测试

覆盖：

- 任务状态转换
- 权限策略判断
- review 生成逻辑

### 2. 适配器层测试

覆盖：

- adapter 可用性判断
- 计划生成结构
- 执行事件输出结构

### 3. 界面层测试

覆盖：

- 关键主流程信息是否渲染
- 关键状态是否能被看到
- 关键行动按钮是否存在

### 4. 构建级验证

覆盖：

- 类型检查
- 构建成功
- 基础打包配置有效

## 三、当前测试重点

现阶段最重要的是守住主流程：

- 任务从 `draft` 走到 `review`
- 权限风险等级正确
- adapter 返回结构符合约定
- UI 能展示主流程与适配器状态

## 四、后续应补充的测试

- 真实 adapter 的契约测试
- 失败恢复测试
- 工作区切换测试
- 持久化读写测试
- 权限拒绝场景测试
- 跨平台差异测试

## 五、测试原则

- 先保主流程，再扩边界场景
- 先测行为，不先测实现细节
- 任何公共契约变化都要有测试证明
- 任何关键 bug 修复都应补回归测试
