# 测试与质量门槛规范

## 一、适用范围

适用于 Mango 的单元测试、组件测试、契约测试、集成测试、跨平台 smoke test、CI 质量门禁与发布前验证。

## 二、测试层级

### 1. 单元测试

- 验证纯领域逻辑、状态机、适配器小单元
- 优先放在各自 package / app 的 `tests/` 或 `src/test/`

### 2. 组件测试

- 验证 renderer 的关键交互与状态呈现
- 按用户行为断言，不耦合实现细节

### 3. 契约测试

- 验证 `@mango/contracts` 的稳定结构
- 验证 IPC channel、DTO、错误模型与共享约束

### 4. 集成测试

- 验证主进程编排、适配器输出、持久化或关键模块联动

### 5. E2E / Smoke

- 验证桌面构建产物、安装包、关键启动路径
- 发布前必须覆盖 Windows、macOS、Linux 基本启动链路

## 三、必须遵守的规则

- 新功能至少对应一种自动化验证
- 共享契约改动必须有契约测试
- 目录或边界改动必须经过 `architecture-check`
- 任何上线前结论都必须基于当次新鲜验证结果

## 四、默认命令入口

- `pnpm lint`
- `pnpm architecture-check`
- `pnpm test`
- `pnpm typecheck`
- `pnpm build`
- `pnpm smoke:desktop`
- `pnpm verify`

## 五、按改动类型要求的最低验证

### 前端改动

- 组件测试
- 类型检查
- 构建

### 本地桌面后端改动

- 单元或集成测试
- 类型检查
- 架构检查
- 构建

### 契约改动

- 契约测试
- 消费方回归
- 文档更新

### 发布链路改动

- CI 验证
- smoke test
- 发布文档同步

## 六、推荐做法

- 优先为共享包和关键流程加快反馈测试
- 在文档中写清“用什么命令证明改动有效”
- 在 CI 中把 lint、typecheck、test、build、smoke 分层执行

## 七、禁止事项

- 没有验证就说“应该可以”
- 只跑单个包测试就宣称全仓通过
- 共享字段变化不回归消费方
- CI 只保留 build，不保留 lint 与边界检查

## 八、评审清单

- 改动对应的测试层级是否齐全
- 验证命令是否可重复
- 是否遗漏失败态、取消态、恢复态
- 是否需要跨平台验证

## 九、常见反模式

- 用快照代替行为验证
- 只测 happy path
- 测试命令分散，团队成员各跑各的
- CI 与本地验证口径不一致
