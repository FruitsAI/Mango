# 版本号与发布通道规范

## 一、适用范围

适用于 Mango 的：

- 应用版本号
- Git tag
- GitHub Releases
- `nightly / beta / stable` 发布通道
- 热修复与回滚命名

## 二、版本治理目标

版本治理不是“发包时随便写个号”，而是确保下面四件事始终一致：

1. 用户看到的版本号
2. 安装包文件名
3. Git tag
4. Release Notes / 更新日志

## 三、版本号规则

### 1. 稳定版基线

Mango 统一采用 `SemVer`：

```text
MAJOR.MINOR.PATCH
```

含义如下：

- `MAJOR`：有明确不兼容变更
- `MINOR`：新增向后兼容能力
- `PATCH`：向后兼容的修复或小幅优化

### 2. 预发布版本

`nightly` 与 `beta` 使用预发布后缀：

```text
0.2.0-nightly.20260324.1
0.2.0-beta.1
```

说明：

- `nightly` 推荐带日期与序号，方便快速定位构建
- `beta` 使用递增序号，便于对外沟通

### 3. Git tag 规则

Git tag 统一使用：

```text
v<version>
```

示例：

- `v0.2.0-nightly.20260324.1`
- `v0.2.0-beta.1`
- `v0.2.0`

## 四、发布通道定义

### 1. `nightly`

- 面向内部验证和高频试错
- 默认从 `main` 自动或半自动产生
- 允许存在已知非阻塞问题
- 必须保证可安装、可启动、主流程可走通

### 2. `beta`

- 面向小规模外部测试用户
- 只接收已经通过主流程回归的能力
- 必须补齐变更说明、已知问题与升级说明

### 3. `stable`

- 面向公开传播与更广泛用户
- 需要通过完整发布检查
- 默认只允许从受控 `release/*` 分支或明确冻结点发布

## 五、分支与通道映射

推荐映射关系：

- `main` -> `nightly`
- `release/<major>.<minor>` -> `beta` / `stable`
- `hotfix/<topic>` -> 合并回 `release/*` 与 `main`

推荐示例：

- `release/0.2`
- `hotfix/windows-installer-timeout`

## 六、安装包与产物命名

安装包文件名统一包含：

1. 产品名
2. 版本号
3. 平台 / 架构
4. 文件扩展名

推荐示例：

- `Mango-0.2.0-beta.1-win-x64.exe`
- `Mango-0.2.0-macos-arm64.dmg`
- `Mango-0.2.0-linux-x64.AppImage`

## 七、发布进入条件

### 1. `nightly` 进入条件

- `pnpm verify` 通过
- 安装包能生成
- 至少完成桌面 smoke 检查

### 2. `beta` 进入条件

- `nightly` 已连续稳定一段时间
- 主流程无阻塞缺陷
- 文档、FAQ、已知问题已更新
- 至少完成目标平台的安装、启动、升级验证

### 3. `stable` 进入条件

- 三端安装包通过验证
- 自动更新、回滚、热修复预案已确认
- Release Notes、下载链接、版本号、Tag 一致
- 发布当天值守与问题收集通道准备完成

## 八、Release Notes 结构

每次发布说明至少包含：

1. 版本号与发布日期
2. 本次重点变化
3. 兼容性说明
4. 升级提示
5. 已知问题
6. 下载入口

推荐模板：

```md
## Mango v0.2.0-beta.1

### 重点变化

### 修复内容

### 升级提示

### 已知问题

### 下载
```

## 九、热修复规则

- 热修复版本默认增加 `PATCH`
- 如果仅针对预发布版本修复，可继续增加预发布序号
- 热修复必须同步回流到主开发线

示例：

- `0.2.0` -> `0.2.1`
- `0.2.0-beta.1` -> `0.2.0-beta.2`

## 十、回滚规则

- 每个 `beta` 与 `stable` 发布都必须能定位到唯一 Git tag
- 回滚时优先回到最近一个已验证的稳定版本
- 如果涉及数据库 migration，必须在发布说明中额外说明是否可无损回滚

## 十一、自动化落地

应通过以下机制约束版本与发布一致性：

- CI 构建安装包
- GitHub Releases 统一产出 changelog 与下载链接
- 发布脚本统一读取同一版本源
- PR 与 Release 文档同步检查

## 十二、推荐做法

- 在里程碑开始前就决定本轮目标通道
- 发布前先冻结 scope，再补说明与验证
- `nightly` 解决“跑不起来”，`beta` 解决“不好用”，`stable` 解决“敢不敢公开发”

## 十三、禁止事项

- Git tag、安装包、Release Notes 三个版本号不一致
- 还没通过 smoke 就发布 `beta`
- 发布当天才写升级说明
- 热修复只修 release 分支，不回流 `main`

## 十四、评审清单

- 版本号是否符合 SemVer
- 发布通道与分支是否匹配
- 安装包命名是否统一
- Release Notes 是否包含升级与已知问题
- 是否具备回滚依据与热修复路径

## 十五、常见反模式

- `nightly`、`beta`、`stable` 只是叫法不同，实际上没有不同门槛
- 为了赶发布临时改版本号，却没同步安装包与 Tag
- 发布成功后才想起写已知问题
